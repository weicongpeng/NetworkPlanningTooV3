# Backend Requirements: 扇区标签设置增强

## Context

我们需要增强地图上扇区标签显示功能。当前实现只有"小区名称"标签可选，字体颜色和大小设置保存后不生效，且缺少频点级别的标签控制。

## Screens/Components

### 扇区标签设置弹窗
**Purpose**: 让用户配置地图上扇区标签的显示内容、样式和显示规则

**Data I need to display**:

1. **标签内容选项列表**
   - 需要展示可选择的标签字段（目前只有"小区名称"）
   - 需要支持多选（用户可以同时显示多个标签字段）
   - 每个选项需要：
     - 显示名称（中文）
     - 对应的字段标识
     - 当前选中状态

2. **字段映射关系**（按网络类型区分）

   LTE 网络类型映射：
   - "小区名称" → "小区名称" 列
   - "基站ID" → "eNodeB标识" 列
   - "下行频点" → "下行链路的中心载频" 列
   - "PCI" → "物理小区识别码" 列
   - "TAC" → "跟踪区码" 列
   - "是否共享" → "是否共享" 列
   - "覆盖类型" → "小区覆盖类型" 列

   NR 网络类型映射：
   - "小区名称" → "小区名称" 列
   - "基站ID" → "gNodeB标识" 列
   - "下行频点" → "填写SSB频点" 列
   - "PCI" → "物理小区识别码" 列
   - "TAC" → "跟踪区码" 列
   - "是否共享" → "是否共享" 列
   - "覆盖类型" → "小区覆盖类型" 列

3. **样式设置**
   - 字体颜色（颜色选择器）
   - 字体大小（数字输入，可能需要范围限制）

4. **扇区级别标签开关**
   - 控制整个扇区的标签是否显示
   - 打开时：所有频点的标签同步打开
   - 关闭时：所有频点的标签同步关闭

5. **频点级别标签控制**
   - 每个扇区的每个频点独立控制
   - 扇区级别打开时，可以单独关闭某个频点
   - 扇区级别关闭时，可以单独打开某个频点

**Actions**:
- **保存标签配置** → 保存用户选择的标签字段、字体样式、显示规则
- **加载已保存配置** → 打开设置时恢复用户之前的配置
- **应用配置到地图** → 保存后立即更新地图上的标签显示

**States to handle**:
- **Initial Load**: 读取已保存的配置或使用默认值
- **Saving**: 保存配置中（可能需要 loading 状态）
- **Error**: 保存失败时需要提示用户
- **Apply**: 保存成功后刷新地图显示

**Business rules affecting UI**:

1. **扇区级别与频点级别的逻辑关系**：
   - 扇区级别 ON → 所有频点标签 ON（用户可单独关闭某些频点）
   - 扇区级别 OFF → 所有频点标签 OFF（用户可单独打开某些频点）
   - 单独切换频点开关不影响扇区级别的开关状态

2. **标签内容多选**：
   - 用户可以同时显示多个字段（如小区名称 + PCI）
   - 标签在地图上可能需要分多行显示或用分隔符分隔

3. **网络类型差异**：
   - LTE 和 NR 的"基站ID"和"下行频点"映射到不同列名
   - 前端需要根据当前数据网络类型使用正确的映射

## Uncertainties

- [ ] 扇区级别标签和频点级别标签的状态应该如何持久化？是否需要在地图数据中包含这些状态？
- [ ] 当用户切换地图数据（如上传新文件）时，标签配置应该如何处理？保留配置还是重置？
- [ ] 多个标签字段同时显示时，在地图上的布局顺序是什么？是否需要让用户控制顺序？
- [ ] 字体大小和颜色是否有合理的默认值和范围限制？
- [ ] 频点级别标签控制的 UI 应该在哪里实现？是在设置弹窗中还是地图上每个扇区的右键菜单中？

## Questions for Backend

- 标签配置应该存储在哪里？前端 localStorage 还是后端需要提供配置存储接口？
- 地图数据中是否已经包含了所有需要的标签字段（基站ID、下行频点、TAC等）？还是需要后端额外提供？
- 后端是否需要提供专门的接口来获取标签字段的值？还是这些数据已经包含在现有的地图数据接口中？
- 频点级别标签的状态应该如何管理？每个扇区-频点组合是否需要唯一的标识符？
