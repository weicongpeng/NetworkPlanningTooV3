- [功能正确性](#功能正确性)

## 概览
本检查清单提供多维度代码审查的详细检查点，用于系统性评估代码质量。

## 功能正确性

### 逻辑检查
- ✓ 条件判断是否完整（if/else、switch/case）
- ✓ 循环终止条件是否正确
- ✓ 递归是否有明确的终止条件
- ✓ 是否处理了所有边界情况（空值、0、负数、最大值、最小值）

### 数据处理
- ✓ 变量使用前是否已初始化
- ✓ 数组/列表访问是否在有效范围内
- ✓ 类型转换是否安全
- ✓ 浮点数比较是否考虑精度问题

### 异常处理
- ✓ 是否有适当的异常捕获和处理
- ✓ 错误信息是否清晰明确
- ✓ 关键操作是否有回滚机制
- ✓ 资源释放是否在finally块或with语句中

## 代码风格

### 命名规范
- ✓ 变量/函数使用有意义的名称
- ✓ 遵循语言命名约定（驼峰式、下划线式）
- ✓ 常量使用全大写
- ✓ 类名使用大驼峰式

### 格式规范
- ✓ 缩进一致（通常2或4空格）
- ✓ 行长度不超过合理限制（通常80-120字符）
- ✓ 适当使用空行分隔逻辑块
- ✓ 运算符两侧有空格

### 注释质量
- ✓ 复杂逻辑有必要的注释说明
- ✓ 函数/类有docstring或注释说明功能
- ✓ TODO/FIXME有说明和责任人
- ✓ 注释与代码保持同步

## 性能效率

### 算法复杂度
- ✓ 时间复杂度是否合理
- ✓ 空间复杂度是否优化
- ✓ 是否有嵌套循环可优化
- ✓ 是否有重复计算可缓存

### 资源使用
- ✓ 是否有不必要的对象创建
- ✓ 数据库查询是否有N+1问题
- ✓ 是否有内存泄漏风险
- ✓ IO操作是否批量处理

### 并发性能
- ✓ 共享资源是否有适当的锁
- ✓ 锁粒度是否合理
- ✓ 是否有死锁风险
- ✓ 是否考虑使用线程安全的数据结构

## 安全性

### 输入验证
- ✓ 是否验证用户输入
- ✓ 是否处理SQL注入风险（使用参数化查询）
- ✓ 是否处理XSS风险（输出转义）
- ✓ 是否处理路径遍历攻击

### 认证授权
- ✓ 敏感操作是否有权限检查
- ✓ 密码是否加密存储
- ✓ 会话管理是否安全
- ✓ API调用是否有认证

### 数据保护
- ✓ 敏感数据是否加密传输
- ✓ 日志是否包含敏感信息
- ✓ 错误信息是否泄露系统细节
- ✓ 第三方依赖是否安全

## 可维护性

### 模块化
- ✓ 函数职责单一，长度合理（通常<50行）
- ✓ 类职责明确，符合SRP原则
- ✓ 是否合理使用设计模式
- ✓ 是否避免深层嵌套（通常<3层）

### 可读性
- ✓ 代码自解释，无需过多注释
- ✓ 避免魔术数字，使用命名常量
- ✓ 避免过长的参数列表（通常<5个）
- ✓ 减少全局变量的使用

### 可扩展性
- ✓ 硬编码值是否提取为配置
- ✓ 接口设计是否灵活
- ✓ 是否便于测试
- ✓ 是否有适当的抽象层

## 常见问题模式

### Python
- 使用 `==` 比较 `None` 而非 `is`
- 使用可变对象作为函数默认参数
- 捕获过于宽泛的异常（如 `except Exception`）
- 未正确处理编码问题

### JavaScript
- 使用 `var` 而非 `let/const`
- 混淆 `==` 和 `===`
- 回调地狱未使用 async/await
- 未处理Promise错误

### Java
- 资源未使用 try-with-resources
- 字符串拼接使用 `+` 而非 `StringBuilder`
- 过度使用静态变量
- 未正确处理并发访问

### 通用
- 过早优化
- 过度设计
- 复制粘贴代码
- 注释掉的大段代码未删除
