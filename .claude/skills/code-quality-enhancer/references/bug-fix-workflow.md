## BUG分类

### 按错误类型
1. **逻辑错误**：代码逻辑不符合需求
   - 特征：程序运行但不产生正确结果
   - 示例：条件判断错误、循环次数不对

2. **语法错误**：代码不符合语言规范
   - 特征：编译或解释时直接报错
   - 示例：缺少分号、括号不匹配、关键字拼写错误

3. **运行时错误**：程序运行时抛出异常
   - 特征：特定条件下崩溃或异常
   - 示例：空指针异常、数组越界、类型错误

4. **性能问题**：程序运行缓慢或资源占用过高
   - 特征：功能正确但效率低下
   - 示例：O(n²)复杂度、内存泄漏、频繁IO

5. **兼容性问题**：特定环境下无法运行
   - 特征：在某些环境正常，其他环境异常
   - 示例：浏览器兼容、版本依赖、平台差异

### 按严重程度
- **P0 致命**：核心功能完全不可用，阻塞用户
- **P1 严重**：主要功能受影响，需紧急修复
- **P2 中等**：部分功能受影响，有变通方案
- **P3 轻微**：小问题，不影响主要功能

## 问题定位

### 信息收集
1. **错误信息**
   - 错误类型和消息
   - 堆栈跟踪
   - 错误代码

2. **复现步骤**
   - 操作序列
   - 输入数据
   - 环境配置

3. **上下文信息**
   - 发生时间
   - 影响范围
   - 最近变更

### 技术手段
1. **日志分析**
   - 查看应用日志
   - 检查系统日志
   - 分析错误日志

2. **代码搜索**
   - 使用 `grep_file` 搜索错误关键词
   - 查找相关函数调用
   - 定位可疑代码段

3. **调试工具**
   - 使用断点调试
   - 打印中间变量
   - 执行单元测试

### 定位策略
1. **二分法**：逐步缩小可疑范围
2. **增量验证**：逐步排除正常部分
3. **对比法**：对比正常和异常情况
4. **时序分析**：按时间顺序追踪事件

## 根因分析

### 分析方法

#### 1. 鱼骨图分析
从人、机、料、法、环五个维度分析：
- 人：开发者、用户操作
- 机：硬件、软件环境
- 料：数据、配置
- 法：代码逻辑、算法
- 环：依赖、外部服务

#### 2. 5Why分析法
连续问5次"为什么"，深入挖掘根本原因：
```
问题：程序崩溃
为什么1？-> 空指针异常
为什么2？-> 变量未初始化
为什么3？-> 初始化逻辑被跳过
为什么4？-> 条件判断错误
为什么5？-> 需求理解偏差（根因）
```

#### 3. 时间线分析
- 系统初始化
- 用户操作开始
- 事件触发
- 错误发生
- 系统恢复

### 根因确认
- ✓ 是否能解释所有现象
- ✓ 修复后问题是否彻底解决
- ✓ 是否有更深层的原因
- ✓ 修复方案是否针对根因

## 修复策略

### 修复原则
1. **最小改动**：只修改必要的代码
2. **影响评估**：评估对其他功能的影响
3. **回归测试**：确保不引入新问题
4. **代码质量**：修复后代码质量不低于原有水平

### 修复方法

#### 1. 直接修复
直接修改错误代码，适用于：
- 逻辑错误
- 边界条件处理
- 简单的拼写错误

#### 2. 防御式编程
添加检查和保护机制，适用于：
- 输入验证
- 异常处理
- 边界保护

#### 3. 重构优化
重构代码结构，适用于：
- 设计缺陷
- 架构问题
- 代码异味

#### 4. 配置调整
修改配置参数，适用于：
- 性能调优
- 环境适配
- 功能开关

### 修复代码示例

#### 示例1：空指针异常
```python
# 问题代码
user = get_user(user_id)
print(user.name)  # user可能为None

# 修复代码
user = get_user(user_id)
if user:
    print(user.name)
else:
    raise ValueError(f"User {user_id} not found")
```

#### 示例2：数组越界
```python
# 问题代码
items = [1, 2, 3]
for i in range(len(items) + 1):
    print(items[i])  # 越界

# 修复代码
items = [1, 2, 3]
for i in range(len(items)):
    print(items[i])
```

#### 示例3：资源泄漏
```python
# 问题代码
f = open('file.txt')
data = f.read()
# 可能因异常未关闭文件

# 修复代码
with open('file.txt') as f:
    data = f.read()
# 自动关闭文件
```

## 验证流程

### 单元验证
1. **语法检查**
   - 使用语言工具检查语法
   - 示例：`python -m py_compile file.py`

2. **静态分析**
   - 使用lint工具
   - 示例：`flake8`, `eslint`, `findbugs`

3. **单元测试**
   - 运行相关单元测试
   - 示例：`pytest`, `jest`, `mvn test`

### 集成验证
1. **功能测试**
   - 执行修复功能的测试用例
   - 验证修复是否生效

2. **回归测试**
   - 运行相关功能的测试套件
   - 确保未引入新问题

3. **性能测试**
   - 对比修复前后的性能
   - 验证性能未退化

### 验收标准
- ✓ 原问题已解决
- ✓ 无新问题引入
- ✓ 相关功能正常
- ✓ 代码质量符合规范
- ✓ 测试通过

## 工具调用示例

### 验证代码语法
```bash
# Python
python -m py_compile script.py

# JavaScript
node --check script.js

# Java
javac Main.java
```

### 运行测试
```bash
# Python
pytest tests/

# JavaScript
npm test

# Java
mvn test
```

### 搜索代码
使用 `grep_file` 搜索相关代码模式
```python
# 搜索可疑函数调用
grep_file(
    grep_content="get_user(",
    grep_path="src/",
    grep_content_mode=1
)
```

### 执行修复验证
```bash
# 编译项目
mvn clean compile

# 运行lint
npm run lint

# 执行测试
pytest test_module.py -v
```

## 注意事项

- 修复前先备份原始代码
- 记录修复过程和决策
- 在测试环境充分验证后再上线
- 对于复杂问题，寻求代码审查
- 更新相关文档和注释
